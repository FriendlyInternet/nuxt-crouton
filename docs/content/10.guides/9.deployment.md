# Deployment Guide

This guide covers deploying nuxt-crouton applications to production environments.

## Deployment Targets

nuxt-crouton apps are built on Nuxt and can deploy to any platform that supports Nuxt:

| Platform | Database | Recommended For |
|----------|----------|-----------------|
| **Cloudflare Workers** (static assets) | D1 (SQLite) | Production apps (recommended) |
| Vercel | Turso/PlanetScale | Vercel-native workflows |
| Netlify | Turso/PlanetScale | Netlify-native workflows |
| Self-hosted | Any SQLite/PostgreSQL | Full control |

::callout{color="primary" icon="i-lucide-star"}
**Recommended**: Cloudflare Workers with D1 provides the best experience for nuxt-crouton apps with edge computing, low latency, and integrated SQLite. Workers **auto-provisions** your D1 database and KV namespace on the first deploy — no manual resource creation, no id-juggling.
::

::callout{color="amber" icon="i-lucide-triangle-alert"}
**Not Cloudflare Pages.** Older versions of this guide (and older crouton apps) used Cloudflare Pages (`wrangler.toml` with `pages_build_output_dir`, `wrangler pages deploy`). The crouton standard is now **Workers with static assets**. If your app is still on Pages, migrate: the setup below replaces it entirely, but note that Pages secrets do **not** carry over — re-set them on the Worker.
::

## Cloudflare Workers Deployment

### Prerequisites

1. **Cloudflare Account** with Workers enabled
2. **Wrangler CLI** (comes as a dev dependency in a scaffolded app; or `npm install -g wrangler`)
3. **Authentication**: `wrangler login`, or set `CLOUDFLARE_ACCOUNT_ID` + `CLOUDFLARE_API_TOKEN` for CI (the token needs **account** Workers Scripts / D1 / KV / R2 Edit, plus **zone** Workers Routes + DNS Edit if you bind a custom domain)

### Step 1: Configure wrangler.jsonc

A freshly scaffolded app (`crouton init`) already ships this. Create or update `wrangler.jsonc` in your app directory — note the bindings are **id-less**: the first deploy auto-provisions them.

```jsonc
{
  "compatibility_date": "2024-09-02",
  "compatibility_flags": ["nodejs_compat"],
  "d1_databases": [
    { "binding": "DB", "database_name": "my-app-db" }
  ],
  "kv_namespaces": [
    { "binding": "KV" }
  ],
  "env": {
    "staging": {
      "d1_databases": [
        { "binding": "DB", "database_name": "my-app-staging-db" }
      ],
      "kv_namespaces": [
        { "binding": "KV" }
      ]
    }
  }
}
```

Key points:

- **No `pages_build_output_dir`** — that's the retired Pages setup.
- **No ids yet** — after the first deploy, run the app's `sync:ids` script (`scripts/sync-wrangler-ids.mjs`, shipped by the scaffolder) to write the provisioned ids back into `wrangler.jsonc`, and **commit them** (remote migrations need them).
- The **`env.staging`** block gives you an isolated staging environment with its **own** D1 + KV — bindings do NOT inherit across envs, so declare them per env.
- `name`/`main`/`assets` are injected by the Nitro preset at build time.

### Step 2: Configure Nuxt

Ensure your `nuxt.config.ts` is configured for Cloudflare:

```typescript
export default defineNuxtConfig({
  // ... your extends

  // CRITICAL: Use db: 'sqlite', NOT database: true
  hub: {
    db: 'sqlite',
    kv: true
  },

  // Disable incompatible features
  croutonAuth: {
    passkeys: false  // Not supported on Cloudflare Workers
  }
})
```

::callout{color="amber" icon="i-lucide-triangle-alert"}
**Common Mistake**: Using `hub: { database: true }` will cause build failures. Always use `hub: { db: 'sqlite' }`.
::

The build uses the Workers preset — the scaffolded `cf:*` scripts already do this:

```bash
NITRO_PRESET=cloudflare_module nuxt build   # output lands in .output/
```

### Step 3: Set Worker Secrets

Your app's secrets live on the **Worker**, not in `wrangler.jsonc`. Worker secrets **persist across deploys**, so this is a one-time bootstrap per environment.

#### Required Variables

| Variable | Description | Example |
|----------|-------------|---------|
| `BETTER_AUTH_SECRET` | Session encryption (32+ chars) | Generate with `openssl rand -base64 32` |
| `BETTER_AUTH_URL` | The deployed URL | `https://my-app.example.com` |

#### Setting via CLI

```bash
# Generate a secure secret
openssl rand -base64 32

# Production Worker
npx wrangler secret put BETTER_AUTH_SECRET
npx wrangler secret put BETTER_AUTH_URL

# Staging Worker (its own separate secret set)
npx wrangler secret put BETTER_AUTH_SECRET --env staging
npx wrangler secret put BETTER_AUTH_URL --env staging
```

For several secrets at once, put them in a JSON object file and use `npx wrangler secret bulk secrets.json` (add `--env staging` for staging). `BETTER_AUTH_URL` must be the **deployed** URL for that environment (not localhost), and each environment has its **own** secret set.

#### Setting via Dashboard

1. Go to [dash.cloudflare.com](https://dash.cloudflare.com)
2. Navigate to **Workers & Pages** → **your Worker** (staging is a separate Worker, e.g. `my-app-staging`)
3. Go to **Settings** → **Variables and Secrets**
4. Add the secrets and click **Save**

### Step 4: First (bootstrap) deploy

The scaffolded scripts chain everything — build, deploy (which auto-provisions D1 + KV), id sync-back, and remote migrations:

```bash
# Staging first (isolated staging Worker + its own D1/KV)
pnpm run cf:staging
# = build → wrangler deploy --env staging → sync:ids → d1 migrations apply --env staging --remote

# Production (deliberate, separate step)
pnpm run cf:deploy
# = build → wrangler deploy → sync:ids → d1 migrations apply --remote
```

After the first deploy of each environment, **commit the ids** that `sync:ids` wrote back into `wrangler.jsonc`.

### Step 5: Database Migrations (subsequent schema changes)

```bash
# Generate migrations (if not already done)
pnpm run db:generate

# Apply to production D1
pnpm run db:migrate:prod
# or: npx wrangler d1 migrations apply my-app-db --remote
```

::callout{color="amber" icon="i-lucide-triangle-alert"}
**Important**: Always run migrations before deploying code that depends on schema changes. (The `cf:deploy` / `cf:staging` chains include the migrate step.)
::

## OAuth Configuration

### Google OAuth

1. Create credentials at [Google Cloud Console](https://console.cloud.google.com/apis/credentials)
2. Add authorized redirect URI: `https://your-domain.com/api/auth/callback/google`
3. Set environment variables:
   ```bash
   npx wrangler secret put GOOGLE_CLIENT_ID
   npx wrangler secret put GOOGLE_CLIENT_SECRET
   ```

### GitHub OAuth

1. Create OAuth app at [GitHub Settings](https://github.com/settings/developers)
2. Set callback URL: `https://your-domain.com/api/auth/callback/github`
3. Set environment variables:
   ```bash
   npx wrangler secret put GITHUB_CLIENT_ID
   npx wrangler secret put GITHUB_CLIENT_SECRET
   ```

## Cloudflare Worker Limitations

Some features are disabled or limited on Cloudflare Workers:

| Feature | Status | Reason | Workaround |
|---------|--------|--------|------------|
| **Passkeys/WebAuthn** | Disabled | `tsyringe` incompatible | Use email/password or OAuth |
| **OG Image Generation** | Optional | Reduces bundle (~4MB) | Use static OG images |
| **Long-running tasks** | Limited | 30s CPU limit | Use Cloudflare Queues |

Configuration for disabled features:

```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  croutonAuth: {
    passkeys: false
  }
})
```

## Deployment Checklists

### First-Time Deployment

- [ ] Create Cloudflare account and `wrangler login` (or set `CLOUDFLARE_ACCOUNT_ID` + `CLOUDFLARE_API_TOKEN`)
- [ ] `wrangler.jsonc` in the Workers shape (id-less bindings, `env.staging` block, no `pages_build_output_dir`)
- [ ] Set `BETTER_AUTH_SECRET` (32+ character secret) on the Worker (per environment)
- [ ] Set `BETTER_AUTH_URL` (the deployed URL, per environment)
- [ ] Configure OAuth providers (if using)
- [ ] Deploy staging: `pnpm run cf:staging` (auto-provisions staging D1 + KV, migrates)
- [ ] Deploy production: `pnpm run cf:deploy` (auto-provisions prod D1 + KV, migrates)
- [ ] Commit the provisioned ids `sync:ids` wrote into `wrangler.jsonc`
- [ ] Test registration and login
- [ ] Verify OAuth flows (if configured)

### Subsequent Deployments

- [ ] If schema changed: `pnpm run db:generate`
- [ ] Deploy: `pnpm run cf:staging` (or `cf:deploy` for production) — the chain migrates for you
- [ ] Verify app functionality

## Troubleshooting

### BETTER_AUTH_SECRET is required

**Cause**: Secret not set on the Worker (each environment has its own set — staging misses it independently of production).

**Fix**:
```bash
openssl rand -base64 32  # Generate secret
npx wrangler secret put BETTER_AUTH_SECRET            # production
npx wrangler secret put BETTER_AUTH_SECRET --env staging  # staging
```
Then redeploy.

### Cannot resolve entry module .nuxt/hub/db/schema.entry.ts

**Cause**: Using `hub: { database: true }` instead of `hub: { db: 'sqlite' }`.

**Fix**: Update `nuxt.config.ts`:
```typescript
hub: {
  db: 'sqlite',  // NOT database: true
  kv: true
}
```

### Remote migrations fail ("couldn't find a D1 DB")

**Cause**: The provisioned `database_id` isn't in `wrangler.jsonc` yet (remote `d1 migrations apply` needs it).

**Fix**:
1. Verify the database exists: `wrangler d1 list`
2. Run the app's `sync:ids` script (part of the `cf:*` chains) and commit the ids it writes back
3. Run: `wrangler d1 migrations apply {app}-db --remote`

### OAuth callback fails

**Cause**: Callback URL mismatch or missing `BETTER_AUTH_URL`.

**Fix**:
1. Ensure `BETTER_AUTH_URL` is set to your deployed URL
2. Verify OAuth provider callback URLs match exactly:
   - Google: `https://your-domain.com/api/auth/callback/google`
   - GitHub: `https://your-domain.com/api/auth/callback/github`

### Build errors with passkeys

**Cause**: Passkey dependencies are incompatible with Workers.

**Fix**: Disable passkeys in config:
```typescript
croutonAuth: {
  passkeys: false
}
```

## Useful Commands

```bash
# Wrangler Management
wrangler login                # Authenticate
wrangler d1 list              # List databases
wrangler kv namespace list    # List KV namespaces
wrangler secret list          # List Worker secrets (add --env staging for staging)
wrangler secret put X         # Add secret X
wrangler tail                 # View live logs

# Database Operations
pnpm run db:generate          # Generate migrations
pnpm run db:migrate           # Apply locally
pnpm run db:migrate:prod      # Apply to production
pnpm run db:migrate:staging   # Apply to staging

# Deployment
pnpm run cf:deploy            # Build + deploy production (+ sync ids + migrate)
pnpm run cf:staging           # Build + deploy staging (+ sync ids + migrate)
```

## Alternative Platforms

### Vercel

1. Configure for Vercel in `nuxt.config.ts`:
   ```typescript
   export default defineNuxtConfig({
     nitro: {
       preset: 'vercel'
     }
   })
   ```

2. Use Turso or PlanetScale for database
3. Set environment variables in Vercel dashboard
4. Deploy via Vercel CLI or Git integration

### Self-Hosted

1. Build the application:
   ```bash
   nuxt build
   ```

2. Run with Node.js:
   ```bash
   node .output/server/index.mjs
   ```

3. Use SQLite file or configure PostgreSQL
4. Set environment variables on your server
5. Use PM2, systemd, or Docker for process management
